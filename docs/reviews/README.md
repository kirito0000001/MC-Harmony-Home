# 模组审查流程

每个模组先进入第 0 阶段，再按结果进入后续审查。第 0 阶段是快速分类，不替代正式结论。

## 第 0 阶段：快速初筛

使用 `tools/quick-mod-intake.ps1 -JarPath <mod.jar>` 提取 NeoForge 元数据与资源信号，记录：

- 模组用途的一句中文介绍。
- 是否存在物品模型、配方、战利品或数据包资源的初步信号。
- 是否像客户端 UI 模组的初步信号。
- 下一步需要的正式审查范围。

输出中的 `likely_client_only_requires_confirmation` 和 `no_item_resource_evidence` 仅用于排队，不能直接决定服务器部署或物品等级。

## 正式阶段

1. 代码与性能审查：仅在有真实收益时修改源码。
2. 服务端部署判断：静态检查，必要时启动专用服务器验证。
3. 汉化审查：中英文齐全，中文使用自然的 Minecraft 术语。
4. Catalogue 介绍与链接：用途、网页和预填 BUG 链接。
5. 物品等级：写入等级数据库或正式标记“不适用”。
6. 运行与发布：安装候选、验证、更新清单、发布 Release。

## 批次验证

- 源码优化后，必须保留构建、针对性回归测试和基础静态检查。
- 不必每完成一个模组就关闭游戏；候选累计到 10 个后统一安装并运行一次客户端验证。
- 崩溃、兼容性风险、共享底层代码、启动阶段修改或用户要求即时验证时，不等待批次，立即单独验证。
- 当前批次状态见 [batch-validation.md](batch-validation.md)。
