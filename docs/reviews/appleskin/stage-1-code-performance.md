# AppleSkin 阶段一：代码与性能审查

检查对象：`AppleSkin 3.0.9`，模组 ID 为 `appleskin`。

## 审查范围与方法

对安装 JAR 的 HUD 叠加层、手持食物查询缓存、提示渲染、同步包和当前客户端配置进行静态检查。使用 JDK 21 的 `javap` 和 `jdeps` 确认缓存节奏、事件入口与网络处理方式。

## 客户端性能路径

- `HeldFoodCache` 以 GUI tick 作为失效条件，同一 tick 内的多个 HUD 层复用食物查询结果；不会在每一帧重复执行 `FoodHelper.query`。
- `OffsetsCache` 同样按 GUI tick 生成食物栏和生命栏偏移，避免每个叠加层重复计算图标位置。
- HUD 绘制仅处理食物与生命图标，提示内容只在悬浮物品时生成；没有轮询线程、计划任务或后台执行器。
- 同步包处理通过 `IPayloadContext.enqueueWork` 回到游戏线程，不跨线程直接修改客户端状态。

## 当前配置

当前 `appleskin-client.toml` 启用了食物提示、饱和度、手持食物恢复预览、副手预览、疲劳条、F3 信息、预计生命恢复和原版动画。该组合优先完整信息展示；其额外成本受上述 tick 缓存限制，正常游玩不构成值得牺牲功能的性能问题。

## 风险与对策

### P2：服务端可选同步的持续 tick 检查

JAR 包含饱和度与疲劳值的可选 payload。若服务端安装 AppleSkin，`SyncHandler` 会在每个 `ServerPlayer` 的每个 tick 比较这两项数值，仅在变化时发送 payload，因此网络带宽不会按 tick 持续增长。它使用两个以 UUID 为键的 `HashMap` 缓存上次值；离线条目不在退出事件中移除，而在该玩家下一次登录时重置。

对策：当前整合包以客户端 HUD 为目标，阶段二将正式评估并优先不把 AppleSkin 复制到服务器目录。这样服务器没有额外 tick 检查和缓存；客户端仍可保留本地食物提示。若未来决定将其作为服务器增强安装，应在长期大型服务器上关注 UUID 缓存规模和每 tick 玩家检查。

### P3：完整 HUD 选项的视觉负担

启用副手预览、健康恢复和动画时，HUD 每 tick 可能额外查询副手食物并绘制更多图标。这是有界的小型客户端开销，不是持续卡顿来源。

对策：保持当前完整信息配置。只有在实际 HUD 冲突或极低端设备掉帧时，才优先关闭 `showFoodValuesHudOverlayWhenOffhand` 或 `showFoodHealthHudOverlay`；不建议修改核心代码。

## 结论

未发现可安全合入且有实际收益的代码优化。AppleSkin 已通过 tick 缓存避免帧率相关的重复食物计算，网络处理也遵守主线程约束；本阶段不修改 JAR、源码或当前配置。

下一阶段是阶段二：正式判断 AppleSkin 的客户端 HUD 与可选服务端同步之间的部署边界，并更新服务器模组筛选记录。该阶段不需要关闭游戏。
