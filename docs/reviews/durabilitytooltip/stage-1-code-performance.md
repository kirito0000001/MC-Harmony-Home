# Durability Tooltip 阶段一：代码与性能审查

检查对象：`durabilitytooltip-1.1.6-neoforge-mc1.21.jar`。

## 代码路径

- 通用入口 `DurabilityTooltip` 只检查 Config Lib 是否存在并初始化配置。
- `DurabilityTooltipClient` 带有 `@EventBusSubscriber(value = Dist.CLIENT, bus = GAME)`；耐久提示只订阅客户端侧、低优先级的 `ItemTooltipEvent`。
- 每次生成物品提示时，会依次判断“只显示原版工具”“模组黑名单”“是否显示满耐久”“是否可损坏”“高级提示模式”，任一不满足便提前返回。
- 通过判断后仅读取 `getMaxDamage` 与 `getDamageValue`，再由三种样式之一追加一行文字、数值或耐久条组件。
- 黑名单配置保存为字符串，但 `lastBlackListConfig` 会缓存最近值。配置未改变时只执行字符串相等判断和 `HashSet.contains`；只有配置变动才重新分割、校验并填充集合。
- JAR 的 Mixin 配置三个数组均为空，访问转换器也是空文件，没有运行时 Mixin 或访问转换逻辑。
- 未发现 payload、packet、network、服务端事件、命令、世界数据或存档读写代码。

## 性能结论

该模组不在每 tick 或每帧主动遍历物品，只在物品提示实际生成时执行。数值与文字样式只构造一行组件；耐久条样式循环构造固定数量的条形字符，开销也受单次提示范围限制。对于当前约 270 个模组的客户端，正常使用下属于可忽略的 UI 开销。

## 风险与对策

1. `P2`：元数据和通用入口会在服务端加载 Config Lib，虽然真正的提示订阅被 `Dist.CLIENT` 限制。
   影响：服务端可能产生一个不参与同步的配置文件；需要在阶段二确认这不是服务端必装或可选增强。
   对策：检查 Config Lib 的 `dontSync` 配置、客户端事件注解与 JAR 网络引用后，再写入服务器筛选记录。

2. `P3`：将样式切为 `BAR` 时，每个物品提示会额外构造一小段耐久条组件。
   影响：在快速浏览创造菜单或 JEI 时，单次提示对象分配略多于数值样式，但没有跨帧缓存或持续任务。
   对策：默认保留 `NUMBERS`，它只生成一行数值；不为这类受提示触发的微小开销引入复杂缓存。需要更简洁界面时可在配置中选择 `TEXT`。

3. `P3`：黑名单配置在被手动改动后会解析长度上限为 100 的逗号分隔字符串。
   影响：首次显示提示时会做一次有限的字符串分割和正则校验；配置稳定后走缓存路径。
   对策：保持黑名单为正常模组 ID 列表，不在运行中反复编辑配置；无需代码优化。

4. `P3`：配置文件的说明文字来自 Config Lib 的英文硬编码注释，而不是语言资源。
   影响：游戏内耐久提示已有中文，但手动打开 TOML 配置时仍会看到英文说明。
   对策：阶段三区分玩家游戏内文本与维护者配置注释，先验证现有 12 个中英文语言键和中文术语；若确有必要，再单独评估可维护的双语配置说明方案。

## 结论与后续流程

阶段一完成。未发现需要直接修复的代码缺陷或高价值性能优化点，当前 JAR 不做改动。下一步进入阶段二，正式确认它的客户端事件与 `dontSync` 配置是否支持 `client_only` 部署结论。
