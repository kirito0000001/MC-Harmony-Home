# Carry On 阶段 0：快速初筛

检查对象：`[搬运] carryon-neoforge-1.21.1-2.2.5.8.jar`。

模组 ID：`carryon`。元数据版本：`2.2.5`。许可证：`GNU LGPLv3`。

## 模组介绍

Carry On 允许玩家在空手时使用“抱起”按键搬运单个方块实体或生物，再将其放到新的位置。常见用途包括移动箱子、熔炉、机器、动物和部分其他模组的可交互方块；搬运对象会暂时显示在玩家手中，并受到黑名单、白名单和自定义拾取条件约束。

它不是普通背包搬运工具。模组会保存被抱起对象的数据、修改玩家与实体交互、发送客户端和服务端数据包，并在放下时重新创建或放置对应对象。

## 快速证据

- JAR 没有 `models/item`、配方、战利品表或物品注册资源。
- `data/carryon` 只包含方块和实体的黑名单、白名单及堆叠标签。
- 公共 Mixin 修改 `Player`、`Entity` 和 `Inventory`；客户端 Mixin 负责模型与输入表现。
- 存在 `ServerboundCarryKeyPressedPacket`、拾取/放置处理器、携带数据管理和客户端同步包。
- 元数据将 NeoForge 与 Minecraft 依赖均声明为 `BOTH`。
- 英文和简体中文语言文件均存在，当前各包含 12 个配置与按键文本键。

## 初步判断

1. 自有物品：高置信度“不存在”。阶段五若阶段一未发现动态注册，可快速标记物品等级不适用。
2. 服务器部署：高置信度“需要客户端与服务器同时安装”。服务端必须验证拾取权限、保存携带数据并实际修改世界；只装客户端无法可靠工作。
3. 世界数据风险：中等。模组会序列化方块实体或实体数据；阶段一应重点检查异常对象、跨维度、死亡/退出和服务器拒绝拾取时的数据恢复。
4. 性能风险：初步较低。正常路径由按键和交互触发，不应持续遍历世界；脚本重载、黑白名单匹配和渲染路径仍需在阶段一确认。
5. 汉化：已有基础简体中文，但只覆盖 12 个配置与按键键。阶段三需确认运行提示、命令、失败原因和高级条件文本是否来自动态文本或缺失语言键。
