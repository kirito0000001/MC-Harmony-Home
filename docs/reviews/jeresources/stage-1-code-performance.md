# JER 阶段一：代码与性能审查

检查对象：`Just Enough Resources 1.6.0.17`。

审查方式：该版本没有可用本地源码工作树，因此以安装 JAR 的 149 个 class、`javap -c -p` 静态反汇编、NeoForge 元数据和配置默认值为证据。结论仅覆盖普通玩家路径，不把未完成的开发/诊断命令当作实际运行负载。

## 范围与结论

- 覆盖模组入口、JEI 注册、兼容层、资源/掉落解析、配置重载、世界卸载、线程池和诊断命令。
- 普通游玩中，JER 把已注册的资源生成、生物掉落、植物、村民与附魔信息整理为内存条目，再交给 JEI 显示；它不在每 tick 重新扫描世界，也不修改世界生成或掉落结果。
- 未发现强制 GC、无限循环轮询、外部进程、网络下载或普通路径中的非受控后台线程。
- 本阶段不建议修改 JAR。它是 JEI 的数据展示扩展，泛化改动战利品解析或注册时序比未测量的性能收益更容易造成错误信息。

## 调用路径证据

### 1. 启动与数据注册

- `jeresources.neoforge.JEResources` 注册 COMMON 配置、配置事件、命令事件和世界卸载事件；客户端和专用服务端按 `Dist` 选择 Proxy。
- `JERAPI.init` 只注入 API 实例，不做全量扫描。
- `CommonProxy.initCompatibility` 会先清空旧的地牢、生物、植物、村民、世界生成条目，再调用 `Compatibility.init` 与附魔筛选。这是启动或资源重载的批处理，不是持续任务。
- `MinecraftCompat.init` 依次登记原版生物、地牢战利品、矿石和植物；模组兼容条目也通过相同注册表合并。

### 2. 战利品表与资源重载

- `LootTableFetcher` 仅持有当前 `ReloadableServerRegistries`，按需从其中取战利品表；没有独立磁盘扫描线程。
- `LootTableHelper.toDrops` 将战利品池、权重、条件和函数转换为 JER 的展示对象。成本集中在登记数据或重载后的重新建立阶段。
- 这解释了 JER 在大型整合包的资源重载时可能有一次性开销，但不会在正常跑图、战斗或 JEI 页面翻页时重复进行全局解析。

## 风险与对策

### 1. 战利品管理器重载

- 风险等级：`P3`，观察项。
- 证据：NeoForge 配置 `disableLootManagerReloading` 默认值为 `false`；配置变更需要重启世界。保持默认时，JER 会在数据包或资源重载后刷新掉落信息。
- 影响：拥有大量数据包和掉落修改模组的实例，执行 `/reload` 或资源重载时可能增加一次性等待；关闭后则会把 JER 显示的掉落信息留在旧状态。
- 对策：保持默认。只有 Spark 采样明确显示 JER 占用显著且用户接受重载后信息可能过期时，才在测试副本比较关闭该项的耗时与准确性。

### 2. 遗留的区块采样线程池

- 风险等级：`P4`，不构成当前实际风险。
- 证据：`ProfilingExecutor` 的确会按 `CPU 核心数 × 2` 创建固定线程池，`Profiler` 也保留了区块采样实现；但 `ProfileCommand` 仅在单人世界、权限等级 4 下暴露 `/jer_profile`，并且本版本 `Profiler.init` 与 `Profiler.stop` 已改为只显示“profiling will be re-added in the future”。
- 影响：普通玩家、服务器和误输入命令都不会启动该线程池或区块采样。
- 对策：不删改死代码，以免破坏将来的上游恢复；不把它当作当前性能问题。若未来版本恢复该命令，再单独评估线程数上限、停止逻辑和世界退出行为。

### 3. 注册表并发与世界卸载

- 风险等级：`P4`，已核查。
- 证据：JER 的展示注册表主要使用 `HashMap`、`LinkedHashSet` 等普通集合；但世界卸载仅在客户端侧清除实体引用，初始化和重载也跟随 NeoForge 生命周期串行执行。未找到后台线程与这些集合并发写入的调用路径。
- 对策：不为假设的竞态引入并发容器。未来若 Spark、崩溃日志或服务端线程转储出现明确竞态证据，再做最小修复。
