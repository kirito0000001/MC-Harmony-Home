# JER 阶段一：代码与性能审查

检查对象：`Just Enough Resources 1.6.0.17`。

## 结论

本阶段不修改 JER JAR。静态反汇编未发现普通游玩路径中的失控线程、强制 GC、轮询任务或会持续改变世界/掉落的行为。

## 核查结果

- 模组初始化只注册通用配置、数据重载处理和 JER API；兼容层会清理并重新建立资源生成、生物掉落、植物、村民与附魔信息的内存注册表。
- 常规信息来自原版/模组已注册的战利品表和资源定义，主要成本发生在启动或资源重载时，不在每 tick 持续扫描世界。
- `jer_profile` 命令表面上包含 `CPU 核心数 × 2` 线程池与区块采样实现，但此版本的 `Profiler.init` 与 `Profiler.stop` 均只返回“profiling will be re-added in the future”的提示，不会启动线程池或扫描区块。
- JER 的资源表和生物掉落表使用普通集合，但初始化与重载回调在既有加载流程中串行执行；未发现可复现的并发共享写入路径。

## 观察项

- 配置 `disableLootManagerReloading` 默认关闭，表示数据包/资源重载时会重新读取战利品表。这可能在 270 模组整合包执行 `/reload` 或重载资源时增加一次性等待，但关闭它会导致数据包修改后的掉落信息过期。
- 对策：保持默认值，不做猜测性优化。若 Spark 采样确认资源重载中 JER 占用显著，再单独比较关闭该项后的正确性与耗时。
